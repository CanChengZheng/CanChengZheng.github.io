题目概要

  1. 海量日志数据，提取出某日访问百度次数最多的那个IP。

分而治之+Hash_map

  1. 搜索引擎会通过日志文件把用户每次检索使用的所有检索串都记录下来，每个查询串的长度为1-255字节，统计最热门的10个查询串。
    问题描述：假设目前有一千万个记录（这些查询串的重复度比较高，虽然总数是1千万，但如果除去重复后，不超过3百万个。一个查询串的重复度越高，说明查询它的用户越多，也就是越热门。），请你统计最热门的10个查询串，要求使用的内存不能超过1G。

分而治之+hash_map+最小堆

  1. 有一个1G大小的一个文件，里面每一行是一个词，词的大小不超过16字节，内存限制大小是1M。返回频数最高的100个词。

分而治之+hash_map+最小堆

  1. 有10个文件，每个文件1G，每个文件的每一行存放的都是用户的query，每个文件的query都可能重复。要求你按照query的频度排序。

分而治之+hash_map+归并排序

  1. 给定a、b两个文件，各存放50亿个url，每个url各占64字节，内存限制是4G，让你找出a、b文件共同的url？

分而治之+hash_set

  1. 在2.5亿个整数中找出不重复的整数，注，内存不足以容纳这2.5亿个整数。

  2. 腾讯面试题：给40亿个不重复的unsigned int的整数，没排过序的，然后再给一个数，如何快速判断这个数是否在那40亿个数当中？

  3. 怎么在海量数据中找出重复次数最多的一个？

分而治之+hash_map

  1. 上千万或上亿数据（有重复），统计其中出现次数最多的钱N个数据。

分而治之+hash_map+最小堆

  1. 一个文本文件，大约有一万行，每行一个词，要求统计出其中最频繁出现的前10个词，请给出思想，给出时间复杂度分析。

题目解析

  1. 上面的 10 道题大致可以分为几种操作方式
    1. 一种是使用N-bitmap对数据进行标记处理（每个数使用 N 个bit进行标记，N取决与实际问题）
    2. 另一种则首先是使用分而治之的思想，将通过hash函数计算并求模的方式，将问题的规模进行缩小，再对问题进行逻辑处理
  2. 其中 6、7 题即可使用2-bitmap的方式进行处理
    1. 第六题：每个数采用2-bitmap进行标记，未出现过标记为00，第一次出现标记为01，多次出现标记为10。因整数类型32位，所以总共可以表示2^32个数值，因而需要 2^32 * 2 bit，即申请总共1GB的内存来进行统计。接下来通过遍历数值，查看bitmap对应的位置数值：如果是00，则变为01；01则变为10；10则维持不变。最后遍历 bitmap，对标记为10的数值进行输出。
    2. 第七题：因为只需要判定是否存在，所以只需要使用1-bitmap进行标记，未出现过标记为0，出现过标记为1。因此需要 2^32 * 1 bit，即申请 512MB 的内存空间进行统计。接下来通过扫描数据，在bitmap中对应的值置为1。接下来扫描bitmap，查找标记为 0 的数值即可。（第七题也有其他更高效的解法，见后续 //TODO）

Activity 的四种启动模式

发表于 2017-08-10

整理自《第一行代码——Android（第2版）》

Activity 的启动模式

  1. 在实际项目中我们可能由于特定的需求为每个 Activity 指定恰当的启动模式。
  2. Activity 的启动模式一共有4种，分别是 standard、singleTop、singleTask 和 singleInstance。
  3. 启动模式的配置：通过在 AndroidManifest.xml 中通过给 标签指定 android:launchMode 属性来配置启动模式。

Standard

  1. standard 是 Activity 默认的启动模式，在不进行显式指定的情况下，所有 Activity 都会自动使用这种启动模式。
  2. 我们知道 Android 是使用返回栈来管理 Activity 。对于使用 standard 模式的 Activity，系统不会在乎这个 Activity 是否已经存在于返回栈中，每次启动都会创建该 Activity 的一个新的实例。每当启动一个新的 Activity，它就会在返回栈中入栈，并处于栈顶的位置。

singleTop

  1. 当 Activity 的启动模式指定为 singleTop，在启动 Activity 时如果发现返回栈的栈顶已经是该 Activity，则认为可以直接使用它，不会再创建新的 Activity 实例。
  2. 而当启动模式为 singleTop 的 Activity 已经存在于返回栈中，但并不位于栈顶，此时再启动该 Activity ，还是会创建新的实例。
  3. 根据第 2 点可以看出，singleTop 模式可能存在的问题是，如果该 Activity 并没有处于栈顶的位置，还是可能会创建多个 Activity 实例。要解决该问题，就需要用到第三种启动模式 singleTask。

singleTask

  1. 使用 singleTask 模式可以很好地解决重复创建栈顶 Activity 的问题。
  2. 当 Activity 的启动模式指定为 singleTask ，每次启动该 Activity 时，系统首先会在返回栈中检查是否存在该 Activity 的实例，如果发现已经存在则直接使用该实例，并把返回栈中在这个 Activity 之上的所有 Activity 都出栈，如果没有发现就会创建一个新的Activity 实例。

singleInstance

  1. 不同于其他 3 种启动模式，指定为 singleInstance 模式的 Activity 会启用一个新的返回栈来管理这个 Acitivity。
  2. 适用的场景：假设我们的程序中有一个 Activity 是允许其他程序调用的，我们想实现其他程序和我们的程序可以共享这个 Activity 的实例。
    1. 如果使用前面 3 种启动模式肯定是做不到的，因为每个应用程序都会有自己的返回栈，同一个 Activity 在不同的返回栈中入栈时必然是创建了新的实例。
    2. 使用 singleInstance 模式就可以解决这个问题，在这种模式下会有一个单独的返回栈来管理这个 Activity ，不管是哪个应用程序来访问这个 Activity ，都共用的同一个返回栈，也就解决了 Activity 实例的问题。

发表于 2017-06-15

整理自《疯狂Java》

Annotation 基本概况

  1. Annotation 是代码里的特殊标记，这些标记可以在编译、类加载、运行时被读取，并执行响应的处理。
  2. 通过使用注解，程序开发人员可以在不改变原有逻辑的情况下，在源文件中嵌入一些补充的信息。代码分析工具、开发工具和部署工具可以通过这些补充信息进行验证或者进行部署。
  3. Annotation 提供了一种为程序元素设置元数据的方法，从某些方面来看，Annotation 就像修饰符一样，可用于修饰包、类、构造器、方法、成员变量、参数、局部变量的声明，这些信息被存储在 Annotation 的“name=value”对中。
  4. Annotation是一个接口，程序可以通过反射来获取指定程序元素的 Annotation 对象，然后通过 Annotation 对象来取得注解里的元数据。
  5. Annotation 不影响程序代码的执行，无论增加、删除 Annotation ，代码都始终如一地执行。如果希望让程序中 Annotation 在运行时起一定的作用，只有通过某种配套的工具对 Annotation 中的信息进行访问和处理，访问和处理 Annotation 的工具统称为 APT(Annotation Processing Tool)。
  6. 使用 Annotation 时要在其前面增加 @ 符号，并把该 Annotation 当作一个修饰符使用，用于修饰它支持的程序元素。

Java 5个基本 Annotation

Java 提供了5个基本 Annotation：

  • @Override
  • @Deprecated
  • @SuppressWarnings
  • @SafeVarargs
  • @FunctionInterface

  1. 限定重写父类方法：@Override

    1. @Override 是用来指定方法覆载的，可以强制一个子类必须覆盖父类的方法。
    2. @Override 的作用是告诉编译器检查这个方法，保证父类要包含一个被该方法重写的方法，否则就会编译出错。
    3. @Override主要是帮助程序员避免一些低级错误，而给后期排错时造成巨大障碍。
    4. @Override 只能修饰方法，不能修饰其他程序元素。

    2. @Deprecated 的作用于文档注释中过的 @deprecated 标记的作用基本相同，但它们的用法不同，前者是 JDK 5 才支持的注解，无须放在文档注释语法（/*../部分）中，而是直接用于修饰程序中的程序单元，如方法、类、接口等。

安装Git

  1. Git 下载地址
  2. 安装 Git 后打开 git bash 界面（在文件夹中右键选择 Git Bash Here），以下所有命令操作全部在此窗口中进行。
  3. 设置 Git 的 user name 和 email
    1. 输入命令：$ git config --global user.name "username" ，这里的username替换为你自己的用户名
    2. 输入命令：$ git config --global user.email "XXX@163.com"，这里同理替换为自己的邮箱
    3. 通过输入命令 $ git config user.name、$ git config user.eamil，可以查看已经设置的用户名和邮箱。

Git SSH key 生成

  1. 首先检查你电脑上的 ssh key 密钥，输入命令 $ cd ~/. ssh
  2. 如果提示：No such file or directory 则说明电脑上没有密钥，继续进行操作；如果没有任何提示，则说明已经有 ssh 密钥，可以跳过下面第3步。
  3. 生成新的 SSH Key
    1. 输入命令 $ ssh-keygen -t rsa -C "haiyan.xu.vip@gmail.com"
      • 注意1：此处的邮箱地址，你要输入自己的邮箱地址
      • 注意2：此处的「-C」的是大写的「C」
    2. 输入第一步的命令，回车一次，然后系统会要求你输入密码
      • 这个密码会在你提交项目时使用，如果为空的话则提交项目时不需要输入密码。这个设置是为了防止别人往你的项目中提交内容。
      • 输入密码的时候没有「*」字样，直接输入即可。
    3. 没记错的话，还需要再按一次回车。如果没有异常信息出现，就成功生成了 ssh key。

补充：

  1. 以后发布博客只需要三步：
    1. hexo new “postname”
      • 新建并编写博文，
    2. hexo g
      • 生成博客
    3. 双击 deploy.sh 脚本文件，输入账号密码即可完成发布。
  2. 如果更换了主题，记得先 hexo clean，再 hexo g 生成博客
  3. _config.yml 文件中的 timezone 配置不能错
    • timezone: zh-CN #能配置成 +08:00 类似的
      上面也有不对的地方，如果设置成 +08:00 后momentjs解析会报异常，需要将timezone配置成时区名称timezone: Asia/Shanghai
  4. 关于域名的绑定，博客现在是绑定在Github的域名上的，可以通过你的username.github.io访问你的博客，如果将来你买了自己的域名，可以将这个域名绑定到Github上，就可以通过你购买的域名访问你的博客了，这是后话了，具体可以自行Google，就不再详述了。

Hexo常用的几个命令

  1. hexo new “postName”
    新建文章，postName是新建的文章的名字，内容就在这里面编写
  2. hexo new page “pageName”
    新建页面
  3. hexo clean
    清除旧主题生成的缓存
  4. hexo generate (hexo g)
    生成静态文件，会在当前目录下生成一个新的名为 public 的文件夹
  5. hexo server (hexo s)
    启动本地web服务，用于博客的预览
  6. hexo deploy (hexo d)
    部署博客到远程服务器上，比如github，heroku等平台

Git 命令 简单整理

发表于 2017-04-19

运行 Git 前的配置

  1. 配置用户信息
    • $ git config –global user.name “ZCC”
    • $ git config –global user.email zcc550169544@163.com
    • 如果用了 –global 选项，那么更改的配置文件就是位于你用户主目录下的那个，以后你所有的项目都会默认使用这里配置的用户信息。如果要在某个特定的项目中使用其他名字或者电邮，只要去掉 –global 选项重新配置即可，新的设定保存在当前项目的.git/config文件里。
  2. 查看配置信息
    • $ git config --list
    • 直接查阅某个环境变量的设定，只要把特定的名字跟在后面即可，
      eg. $ git config user.name
  3. 获取帮助
    • 想了解 Git 的各种工具怎么使用，可以通过输入命令阅读它们的使用帮助，格式：$ git help
      eg. $ git help config

Git 基础

取得项目的 Git 仓库

  1. 从当前目录初始化
    • $ git init
  2. 从现有的仓库克隆
    • $ git clone [url]
      eg. 克隆Ruby语言的 Git 代码仓库：$ git clone git://github.com/schacon/grit.git
    • 克隆的时候，自己定义要新建的项目目录名称，可以在上面的命令最后指定：
      $ git clone git://github.com/schacon/grit.git mygrit
    • Git 支持许多数据传输协议，详情见Pro git P14 ​

记录每次更新到仓库

  1. 检查当前文件状态
    • 要确定哪些文件当前处于什么状态：$ git status
  2. 跟踪新文件
    • $ git add README
  3. .gitignore格式规范
    • Pro git P18
  4. 查看具体修改了什么地方
    • $ git diff
    • 详情见Pro git P19
  5. 提交更新
    • $ git commit
      这种方式会启动文本编辑器来输入本次提交的说明。
    • $ git commit -m
      使用 -m 参数后跟提交说明的方式，在一行命令中提交更新。
  6. 移除文件
    • $ git rm
    • 详情见Pro git P19
  7. 移动文件
    • $ git mv

查看提交历史

详情见Pro git P24

  1. 回顾提交历史
    • $ git log
    • –p 选项展开显示每次提交的内容差异
    • -2 仅显示最近的两次更新
      $ git log –p -2
    • –stat 仅显示简要的增改行数统计
    • –pretty 可以指定使用完全不同于默认格式的方式展示提交历史，详情见Pro git P26

撤销操作

详情见Pro git P29

  1. 修改最后一次提交
    • $ git commit –amend
  2. 取消已经暂存的文件
    • $ git reset HEAD …
  3. 取消对文件的修改
    • $ git chechout – …

远程仓库的使用

  1. 查看当前的远程库
    • $ git remote，列出每个远程库的简短名字
    • -v 选项，显示对应的克隆地址
  2. 添加远程仓库
    • $ git remote add [shortname] [url]
  3. 从远程仓库抓取数据
    • $ git fetch [remote-name]
    • fetch 命令只是将远端的数据拉到本地仓库，并不自动合并到当前的工作分支，当你准备好了，才能手工合并。
  4. 自动抓取数据
    • $ git pull
    • 详情见Pro git P33
  5. 推送数据到远程仓库
    • $ git push [remote-name] [branch-name]
  6. 查看远程仓库信息
    • $ git remote show [remote-name]
  7. 远程仓库的删除和重命名
    • 重命名：$ git remote rename pb paul
    • 删除：$ git remote rm paul

打标签

  1. 显示已有的标签
    • $ git tag
  2. 创建含附注的标签
    • $ git tag -a v1.4 -m ‘my version 1.4’
    • -a 指定标签名字
    • -m 指定对应的标签说明
    • $ git show v1.4 查看相应标签的版本信息
  3. 创建轻量级标签
    • $ git tag v1.4
